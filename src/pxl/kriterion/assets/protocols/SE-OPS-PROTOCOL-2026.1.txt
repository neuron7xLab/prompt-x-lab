/EXPLICIT_SCORE_AGGREGATION
A) ARTIFACT → VALIDATION
Artifacts MUST be CanonicalArtifactJSON and schema-validated.
Fingerprint required; missing fingerprint => G0_INTEGRITY FAIL.

Fingerprint:
- files: sha256(canonicalized_bytes)
- repos/paths: sha256(repo_url + commit_hash + path)
- PRs: sha256(repo_url + pr_number + merge_commit_hash)

B) TASK → TASK_SCORE
Task VALID iff:
- required artifact types present + schema-valid + fingerprinted
- >=1 UNIQUE_PRIMARY artifact (anti-gaming)
- reproducibility present where required

Independent review is not mandatory for this level, but:
- HIGH EC requires at least one reviewer OR immutable record (immutability_ref) for changes affecting production.

TaskScore:
- INVALID => 0
- else TaskScore = min(RawScore, EC_CAP(task_EC)) then penalties
Penalties:
PenaltyPoints = min(2, #SEVERE_NEG_TASK) + min(1, #MEDIUM_NEG_TASK)
TaskScore = max(0, TaskScore - PenaltyPoints)

task_EC:
- HIGH: schema-valid + reproducibility where required + clear timeline + (if prod change) rollback notes + immutable record
- MED: schema-valid but missing one element of HIGH criteria
- LOW otherwise

C) DOMAIN → DOMAIN_SCORE
DomainScore = Σ(TaskWeight * TaskScore)/100
DomainEC:
- HIGH: >=2 VALID tasks AND >=60% domain-weighted tasks have task_EC=HIGH
- MED: >=1 VALID task with schema-valid evidence
- LOW otherwise
DomainScore capped by EC_CAP(DomainEC)

D) TOTALS
OpsScore = Σ(OD_weight * OD_score)/100
ToolingScore = Σ(TF_weight * TF_score)/100
CompositeScore = 0.80*OpsScore + 0.20*ToolingScore

Classification:
- FAIL if any gate fails
- SECURITY_ENGINEER_READY if Composite >= 3.0 and MUST_HAVE >= 2.8
- STRONG_SECURITY_ENGINEER if Composite >= 3.6 and MUST_HAVE >= 3.2 and >=2 MUST_HAVE domains with DomainEC>=MED
- SENIOR_POTENTIAL if Composite >= 4.1 (informational)

/ANTI-GAMING_RULES
Definitions:
- PRIMARY_EVIDENCE: artifact used to satisfy required types for a task
- UNIQUE_PRIMARY: fingerprint not used as PRIMARY_EVIDENCE for any other task
- DUPLICATE_ARTIFACT: same fingerprint across tasks/domains

Rules:
AG1 Each task requires >=1 UNIQUE_PRIMARY else INVALID.
AG2 fingerprint may be PRIMARY_EVIDENCE for at most ONE task; reuse => later task INVALID.
AG3 Secondary references allowed but do not satisfy requirements and do not raise EC.
AG4 Semantic-duplicate detection => semantic_duplicate=true treated as DUPLICATE; intentional => DISQUALIFIER D2.

/ARTIFACT VALIDATION SCHEMAS (EXPLICIT)
General validation (ALL):
- title, pointer non-empty; timestamp_range valid; fingerprint present; provenance.author+role present;
  confidentiality.redaction_statement present; schema_validation.passed=true.
- For HIGH EC in production-affecting artifacts: provenance.immutability_ref must be present.

CONTROL_CHANGE_RECORD:
- system/component; change summary; environment (prod/stage); rollout plan; rollback plan; safety checks; approvals or ticket id.
Validation:
- if environment=prod => rollback required and immutability_ref required for HIGH EC.

AGENT_DEPLOYMENT_RECORD:
- agent/tool name+version; target scope; deployment method; verification steps; rollback/uninstall steps.
Validation:
- must include verification steps and at least one failure mode.

CONNECTOR_CONFIG_RECORD:
- data source; auth method; mapping/normalization; reliability notes; backpressure/limits; logging.
Validation:
- must link to DASHBOARD_SNAPSHOT or METRICS_SNAPSHOT for HIGH EC.

ALERT_TRIAGE_LOG:
- alert id/type; time received; triage steps; outcome (benign/TP/needs escalation); evidence pointers.
Validation:
- must include decision rationale and next action.

ESCALATION_RECORD:
- reason; severity; who paged/notified; handoff notes; evidence pointers.
Validation:
- must include timestamped handoff and owner.

RUNBOOK:
- triggers; preconditions/access; steps>=8; decision points>=2; safety checks+rollback; escalation; evidence preservation.
Validation:
- explicit no-destroy-evidence guidance unless exception documented.

INCIDENT_TICKET:
- incident id; severity; timeline; actions taken; comms notes; final disposition.
Validation:
- must link to at least one ESCALATION_RECORD and one ALERT_TRIAGE_LOG (or justified absence).

VULN_SCAN_REPORT:
- sources/tools; scope; time window; results summary; raw counts by severity; known limitations.

VULN_TRIAGE_REPORT:
- prioritization method (CVSS+context); SLAs; owners; remediation plan; exceptions.
Validation:
- accepted items must link to approvals (SECURITY_REVIEW_NOTES or ticket) and include expiry.

PATCH_VALIDATION_REPORT:
- patch applied; verification method; before/after evidence; regression checks; rollback plan if failed.
Validation:
- includes at least one verification command or test output for HIGH EC.

CLOUD_FINDINGS_REPORT:
- findings source; affected resources; severity; remediation steps; validation evidence.
Validation:
- must include scope definition and remediation validation for HIGH EC.

BASELINE_HARDENING_CHECKLIST:
- baseline controls list; current status; deviations + rationale; remediation plan.

METRICS_SNAPSHOT:
- metric names; method/query; time window; baseline+after if improvement claimed; limitations.

DASHBOARD_SNAPSHOT:
- purpose; signals; thresholds; time window; operator notes.

SECURITY_REVIEW_NOTES (lightweight for this level):
- date; scope reviewed; findings>=2; decisions/recommendations; reviewers (optional).
Validation:
- if used to justify accepted risk => approvals non-empty and expiry recorded elsewhere (VULN_TRIAGE_REPORT or exception).

- At least 2 distinct triage artifacts (ALERT_TRIAGE_LOG) and 1 escalation chain (ESCALATION_RECORD + INCIDENT_TICKET)
