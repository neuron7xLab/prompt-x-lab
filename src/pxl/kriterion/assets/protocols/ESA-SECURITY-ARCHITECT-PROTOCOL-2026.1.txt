/EXPLICIT_SCORE_AGGREGATION
A) ARTIFACT → VALIDATION
Artifacts MUST be represented as CanonicalArtifactJSON and validated against ArtifactTypeSchema.
Fingerprint required; missing fingerprint => G0_INTEGRITY FAIL.

Fingerprint:
- files: sha256(canonicalized_bytes)
- repos/paths: sha256(repo_url + commit_hash + path)
- PRs: sha256(repo_url + pr_number + merge_commit_hash)

B) TASK → TASK_SCORE
Task VALID iff:
- required artifact types present + schema-valid + fingerprinted
- >=1 UNIQUE_PRIMARY artifact (anti-gaming)
- if task_EC=HIGH and domain is MUST_HAVE => independent review requirement satisfied

TaskScore:
- INVALID => 0
- else TaskScore = min(RawScore, EC_CAP(task_EC)) then apply penalties
Penalties:
PenaltyPoints = min(2, #SEVERE_NEG_TASK) + min(1, #MEDIUM_NEG_TASK)
TaskScore = max(0, TaskScore - PenaltyPoints)

task_EC:
- HIGH: required artifacts schema-valid + independent reviewer + (where applicable) risk metrics/bounds + governance trace
- MED: schema-valid but missing one of the above
- LOW otherwise

IndependentReviewerRequirement (mechanized):
- For HIGH EC in MUST_HAVE domains, at least one included artifact MUST have
  provenance.reviewers containing >=1 reviewer != author and reviewer_roles[index] != author.role
  AND reviewer appears in approvals OR provenance.immutability_ref indicates review record.
If missing => task_EC downgraded to MED.

C) DOMAIN → DOMAIN_SCORE
DomainScore = Σ(TaskWeight * TaskScore)/100
DomainEC:
- HIGH: >=2 VALID tasks AND >=60% of domain-weighted tasks have task_EC=HIGH
- MED: >=1 VALID task with schema-valid evidence
- LOW otherwise
DomainScore capped by EC_CAP(DomainEC)

D) TOTALS
ArchitectureScore = Σ(AD_weight * AD_score)/100
IntegrationScore = Σ(TD_weight * TD_score)/100
CompositeScore = 0.65*ArchitectureScore + 0.35*IntegrationScore

Classification:
- FAIL if any gate fails
- LEAD_ARCH_READY if Composite >= 3.7 and MUST_HAVE >= 3.3
- ENTERPRISE_ARCH_READY if Composite >= 4.1 and MUST_HAVE >= 3.7 and >=3 MUST_HAVE domains with DomainEC=HIGH

/ANTI-GAMING_RULES
Definitions:
- PRIMARY_EVIDENCE: artifact used to satisfy required types for a task
- UNIQUE_PRIMARY: fingerprint not used as PRIMARY_EVIDENCE for any other task
- DUPLICATE_ARTIFACT: same fingerprint across tasks/domains

Rules:
AG1 Each task requires >=1 UNIQUE_PRIMARY else INVALID.
AG2 fingerprint may be PRIMARY_EVIDENCE for at most ONE task; reuse => later task INVALID.
AG3 Secondary references allowed but do not satisfy requirements and do not raise EC.
AG4 Semantic-duplicate detection: trivial edits => semantic_duplicate=true => treat as DUPLICATE.
Intentional semantic duplication triggers DISQUALIFIER D3.
AG5 MUST_HAVE independence: HIGH EC requires independent reviewer; repeated self-approval patterns cap DomainEC to MED.

/ARTIFACT VALIDATION SCHEMAS (EXPLICIT)
General validation (ALL):
- title, pointer non-empty; timestamp_range valid; fingerprint present; provenance.author+role present;
  confidentiality.redaction_statement present; schema_validation.passed=true;
- provenance.immutability_ref required for HIGH EC eligibility.

SECURITY_REVIEW_NOTES (strengthened):
- date; scope reviewed (artifact ids); findings>=3; decisions/recommendations; accepted risks (if any) + rationale;
  reviewers + roles.
Validation:
- reviewers length>=1
- For HIGH EC in MUST_HAVE: independent reviewer present (reviewer!=author and reviewer_role!=author.role)
  and corroborated via approvals or immutability_ref.

ADR:
- context; decision (single); alternatives>=2; consequences; security invariants>=3; threat model linkage;
  migration/rollback; owners/approvals; acceptance criteria>=1 measurable; linkages>=1.

DESIGN_DOC:
- problem; goals/non-goals; architecture overview; data flows; security considerations; operational plan;
  compatibility/migration; failure modes>=3; references >=1 THREAT_MODEL or ADR.

REFERENCE_ARCH:
- scope + assumptions; target state; standard patterns; control placement; trust boundaries; integration guidance;
  exceptions model; rollout/adoption guidance.
Validation:
- references >=1 CONTROL_MAP and >=1 THREAT_MODEL (or justified none).

THREAT_MODEL:
- assets; actors; trust boundaries; entry points; threats>=10 with severity+likelihood bounds;
  mitigations mapped; residual risk; detection/control mapping >=5 or justified NONE.

ATTACK_TREE:
- root; depth>=3 or nodes>=12; node annotations incl. control/detection/assumptions.

RISK_REGISTER:
- top risks>=10; each has: description, impacted assets, likelihood bounds, severity/impact, owners,
  mitigation plan, residual risk, review cadence, decision status.
Validation:
- each risk links to at least one mitigation artifact_id (ADR/CONTROL_MAP/MIGRATION_PLAN) or accepted-risk approval.

CONTROL_MAP:
- mapping threats→controls→owners→evidence pointers.
Required fields per row:
  threat_id, control_id, control_type(prevent/detect/respond), enforcement_point,
  evidence_artifact_ids[], effectiveness_notes.
Validation:
- >=15 mappings OR justified smaller scope; must cover all “HIGH severity” threats from THREAT_MODEL.

BLUEPRINT types (CLOUD/IDENTITY/APPSEC/NETSEC/DATA_PROTECTION):
- domain scope; baseline state; target state; trust boundaries; key controls; integration points;
  failure modes; migration plan reference; exceptions model.
Validation:
- references >=1 THREAT_MODEL and >=1 CONTROL_MAP.

MIGRATION_PLAN:
- phases>=3; dependencies; rollback per phase; comms plan; risks; success metrics.
Validation:
- includes metrics + owners.

EXCEPTION_RECORD:
- exception description; scope; expiry date; compensating controls; approval trail.
Validation:
- approvals non-empty; expiry present.

METRICS_SNAPSHOT / DASHBOARD_SNAPSHOT:
- metric names; method; time window; baseline+after (if claiming improvement); limitations.
Validation:
- cannot claim improvement without baseline+after.

TEST_RESULTS (optional, but boosts EC in integration domains):
- plan; environment; coverage; results; repro; negative test case>=1
If present and reproducibility.required=true => commands non-empty.

EXEC_SUMMARY (optional):
- risk framing; options; recommendation; dependencies; timeline.
